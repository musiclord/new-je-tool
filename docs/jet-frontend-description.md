# JET 前端介面規格書

本文件是**執行期前端**（`src/JET/JET/wwwroot/`）的行為與流程權威描述，涵蓋畫面結構、使用者流程、互動狀態與資料契約。它不是 HTML / CSS 的逐行說明，也不是 action contract 的替代品。換言之，即使原始程式不存在，工程師也應能依本文件重建出功能等價的前端。

> 2026-06-11 起本文件以六步驟執行期前端為準（原始模板 `docs/jet-template.html` 降為視覺參考）；規則一律使用 `docs/jet-guide.md` §4 命名登錄表的具體名稱，V/R/A 代號已退役。

## 2. 介面原則

JET 前端是運行於 WebView2 中的審計工作流程 UI。它的職責是收集使用者輸入、呈現流程狀態、組合使用者的選擇，並透過 action contract 將請求交由後端處理。

前端是一層「薄」的呈現與協調層，**不得**持有任何權威性的審計邏輯，包括：

- 審計規則與 SQL
- 資料匯入的欄位標準化（投影）
- 資料驗證、預篩選、進階篩選的計算邏輯
- 工作底稿匯出的產製邏輯

上述邏輯一律由後端負責；前端僅負責觸發、傳遞參數與呈現結果。

### 詞彙規範

- 規則名稱一律用命名登錄表的中文名（完整性測試、期末後核准…），不得出現 V1/R8 等已退役代號。
- 對使用者顯示「標準化」（如「標準化後分錄」「已標準化 N 列」），不用實作詞「投影」；「測試母體」「匯入原貌」為正當審計詞彙。
- 不對使用者顯示技術內部詞（AST、SQL、payload、metadata 等）。

## 3. 模組結構

JavaScript 採模組化結構，載入順序見 `index.html`：

| 模組 | 職責 |
|:---|:---|
| `js/jet-api.js` | 唯一的 WebView2 傳輸層（`JetApi.*` typed facade；`SUPPORTED_ACTIONS` 清單鏡像 manifest） |
| `js/state.js` | 輕量 UI 狀態（`JetStore`）：步驟、專案、匯入/配對/規則執行/篩選狀態；**不存 GL/TB row 集合** |
| `js/ui-core.js` | 共用核心（`JetUi`）：契約鏡像常數（欄位、模式、條件型別與分組、預篩選鍵、分類選項）、閘門模型、統一頁尾、步驟/mock 註冊表、專案載入、全域欄位驗證提示（`setFieldError`/`clearFieldError`） |
| `js/steps/*.js` | 每步驟一個渲染模組，自行向 `ui-core` 註冊 |
| `js/data-preview.js` | 資料預覽折疊面板（正式版功能） |
| `js/dev-panel.js` | 開發者面板（僅 Debug 組建顯示） |
| `js/app.js` | 殼層與啟動（專案選擇畫面、步驟導航、訊息區渲染） |

## 4. 應用程式外殼

| 區域 | 必要元素 | 用途 |
|:---|:---|:---|
| 頁首 | 產品名稱、bridge 狀態、案件/客戶/步驟 badge、「回專案選擇」、「儲存並結束」 | 顯示 host 連線狀態、目前案件與流程位置 |
| 專案選擇畫面 | 單一專案列表（`project.list`）、「新增專案」列 | 啟動入口；載入續作（resume）、刪除專案 |
| 左側步驟導航 | 六個步驟按鈕 | 未解鎖步驟停用並以 tooltip 列出缺漏條件；切換步驟即時保存位置 |
| 主畫面區 | 同一時間僅顯示一個步驟面板 | 呈現目前流程步驟 |
| 統一步驟頁尾 | 上一步/下一步、閘門提示 | 閘門未滿足時「下一步」停用並說明缺什麼；第 0 步（建立案件）無上一步按鈕 |
| 訊息區 | 全域訊息流（info/warn） | 各 action 的成功摘要與錯誤訊息 |
| 資料預覽面板 | 資料集下拉、重新整理、有界資料表（≤50 列＋總列數＋概況統計） | 檢視已匯入／已標準化的資料集（**正式版功能**，`query.dataPreview`） |
| 開發者面板 | 唯讀資料庫檢視 | **僅 Debug 組建**：依 `system.ping.devToolsEnabled` 顯示；Release 後端不註冊 `dev.db.*` |
| 測試案件按鈕 | 每步驟「套用測試案件」 | dev 輔助：走正式 file-based 管線的 deterministic demo fixture |

**頁首。** 「回專案選擇」只在 workflow 視圖顯示,點它會先 `project.saveProgress` 再切回 picker。「儲存並結束」結束前也是先 `project.saveProgress` 再 `host.exitApp`。

**統一步驟頁尾。** 第 0 步沒有上一步按鈕,回專案選擇改由頁首提供。

**專案選擇畫面。** 專案列表每列顯示客戶名稱、案件編號、provider 標籤（SQLite／SQL Server）與上次開啟時間。整列可點即開啟,沒有獨立的開啟鈕。每列尾端有一個低調的刪除鈕:點下去先彈確認框（顯示名稱並提示無法復原）,確認後送 `project.delete`,連同資料庫一併刪除。最底的「新增專案」列觸發建立流程。訊息面板在此視圖同樣可見。

**訊息區。** picker 與 workflow 兩視圖共用同一面板:單一 DOM、同一訊息來源與收合狀態。面板預設收合成右側窄欄,只顯示一個垂直的「訊息」標籤和未讀徽章;若收合期間有警告（warn）訊息,徽章會轉成警示色。點一下展開成 300px 的面板。各 action 的成功摘要與錯誤訊息都會進到這裡,匯入與標準化的完成訊息還會附上耗時秒數。專案選擇畫面的事件（如 `project.list`/`create`/`delete`/`load` 失敗）也即時可見,不再靜默。訊息會持久化於專案資料庫（`log.append`,每專案留最近 500 則）,`project.load` 後再以 `log.recent` 還原歷史。沒有 active project（也就是停在專案選擇畫面）時,訊息只存在前端記憶體。

**資料預覽面板。** 提供六個資料集:GL/TB 來源原貌、GL 標準化後分錄（測試母體,附金額/日期概況）、TB 標準化後餘額、科目配對、授權編製人員清單。配對與篩選步驟有換位按鈕;匯入步驟的科目配對卡與授權編製人員卡各有「預覽」鈕。這個底部面板只服務已匯入／已標準化的資料集;匯入前的逐來源預覽（表頭＋前 10 列）在匯入精靈內,見 §7。

## 5. 流程總覽（六步）

```mermaid
flowchart LR
  picker["專案選擇"]
  s0["0. 建立案件"]
  s1["1. 匯入資料"]
  s2["2. 欄位配對"]
  s3["3. 資料驗證與測試"]
  s4["4. 進階條件篩選"]
  s5["5. 匯出底稿"]

  picker --> s0 --> s1 --> s2 --> s3 --> s4 --> s5
```

閘門模型（`ui-core.stepGate`；線性流程，進入第 n 步須滿足沿路所有閘門）：

| 步驟 | 進入條件 | 完成（推進下一步閘門）條件 |
|:---|:---|:---|
| 0 建立案件 | 預設入口 | 已建立或載入案件 |
| 1 匯入資料 | 有案件 | GL 與 TB 皆已匯入 |
| 2 欄位配對 | GL/TB 已匯入 | GL 與 TB 配對皆已提交 |
| 3 資料驗證與測試 | 配對已提交 | 已執行資料驗證 |
| 4 進階條件篩選 | 已執行驗證 | 已保存至少一個篩選情境 |
| 5 匯出底稿 | 已保存情境 | （匯出里程碑，佔位） |

使用者所在位置以 `project.saveProgress` 即時保存（允許倒退）；`project.load` resume 到上次位置，若資料不再滿足閘門則退回最近可達步驟。

## 6. 步驟 0：建立案件

- 輸入案件基本資料（**案件名稱**、案件編號、客戶名稱、操作人員、產業別、查核期間、期末財報準備日）→ `project.create`。**案件名稱**亦作為專案資料夾名與內部 projectId（取代雜湊;經字元白名單驗證、同名擋下;前端必填,只有未填的程式化建立才回退 GUID）。
- **資料庫 provider 選擇**（下拉）：SQLite（本機檔案，預設）／SQL Server（共用實例）；隨 payload 送 `databaseProvider`，**建立後不可變更**（選 SQL Server 需後端已設定 `JET_SQLSERVER_CONNECTION`，否則建立時得到 `sql_server_not_configured`）。
- 已有案件時顯示唯讀摘要（含所選 provider）。建立後自動 `project.load` 進入工作流程。

## 7. 步驟 1：匯入資料

四張卡片：GL（總帳明細）、TB（試算表）、科目配對、日期維度（假日/補班日）。

### GL / TB：多來源匯入精靈

一個資料集 = 一個批次，可由多個檔案或多個工作表合併組成（guide §3.1.4）。匯入卡片是**三個互斥狀態**的狀態機，入口鈕與〔開始匯入〕**永不同框**：

| 卡片狀態 | 顯示內容 | 可用動作 |
|:---|:---|:---|
| 空狀態（尚未匯入） | 「尚未匯入。可由多個檔案或多個工作表合併成一個資料集。」 | 僅〔選擇來源檔〕 |
| 已匯入摘要 | 已匯入列數/欄數/來源數、來源清單（檔名、工作表膠囊、編碼/分隔符、列數、時間） | 〔加入來源〕（`append`）、〔重新匯入〕（`replace`）；**不顯示**〔開始匯入〕 |
| 匯入工作區 | 模式橫幅＋待匯入清單＋進度條 | 〔選擇來源檔／再加入檔案〕、〔開始匯入（n）〕、〔取消〕 |

空狀態的〔選擇來源檔〕會直接以 `replace` 模式開啟工作區,並立即彈出檔案對話框,不需要第二次點擊。進入匯入工作區後,摘要面與入口鈕都不再渲染。

模式橫幅標示本次匯入的模式：`replace`（重新匯入）是破壞性操作，用警示色，說明現有的 N 列將被新來源取代、欄位配對需重做、先前的測試與篩選結果也會清除；`append`（加入來源）是藍字，說明新資料會附加到現有的 N 列、欄位配對同樣需重做、先前的測試與篩選結果也會清除；若是首次建立則為藍字。

工作區流程：

| 觸發 | UI 行為 | 後端 action |
|:---|:---|:---|
| 選擇來源檔（可多選，可累加） | 原生多選對話框，逐檔附加到待匯入清單 | `host.selectFiles` |
| 逐檔預覽（待匯入清單） | xlsx 逐工作表列出（可勾選）、CSV 顯示偵測到的編碼/分隔符（可下拉覆寫） | `import.inspectFile` |
| 逐來源〔預覽 ▸〕 | 每列展開「表頭＋前 10 列原貌」的有界小表（≤10 列） | `import.previewFile` |
| 開始匯入（n） | 依序匯入勾選來源；失敗即中止、已成功來源保留；勾選歸零時停用 | `import.gl.fromFile` / `import.tb.fromFile` |
| 取消 | 關閉工作區回到原狀態（空／摘要），無副作用 | — |

逐檔預覽時,xlsx 的每張工作表會標出「N 欄・約 X 列」。這個列數是 `rowCountEstimate` 推估值,僅供顯示。

逐來源預覽展開的小表絕不載入完整母體,最多 10 列。表上附一句提示:「最上方一列是被當成欄名的標頭；若它看起來是資料而非欄名，代表這份檔案可能沒有標頭列」——用意是讓使用者人工判讀**無標頭檔**。預覽結果按來源快取;CSV 改了編碼或分隔符後,該快取失效並重抓;展開失敗則收合。

開始匯入時依序匯入勾選的來源:第一個依工作區當下的模式（`replace` 或 `append`）,其後一律 `append`。

匯入期間訂閱 `import.progress` 事件顯示細進度條（`JetApi.on`,即 manifest 的「Host→Web 事件」）。xlsx 顯示成「已寫入 N 列（約 P%）」,百分比是已讀列數 ÷ inspect 估計值,上限 99%,完成與否以 response 為準;CSV 沒有估計值,只顯示已寫入列數。精靈結束（無論成功或失敗）即解除訂閱。

匯入成功後顯示來源清單（檔名、工作表膠囊、編碼/分隔符、列數、時間）。重新匯入或加入來源完成、卡片收回摘要面時,卡片上會短暫出現一個成功態（重新匯入是「剛剛重新匯入」、加入來源是「已加入來源」,後面接完成時間）,幾秒後淡回常態的已匯入摘要。這個確認訊號就在卡片上,不會增加訊息面板的文字量。匯入（replace 或 append）會使已提交配對失效:後端清除,前端同步標記,見步驟 2 的狀態模型。

### 科目配對

格式固定三欄（科目代號、科目名稱、標準化分類），匯入即生效、整份替換（`mode:"append"` 不支援）。`host.selectFile`（.xlsx/.csv）→ `import.accountMapping.fromFile`。匯入後解鎖「未預期出現之特定借貸組合」預篩選與「科目配對分析」篩選條件。分類非法 → `projection_failed`，訊息列出列號與原值。

### 日期維度

假日與補班日清單（demo 走 `import.holiday` / `import.makeupDay`；正式匯入介面屬後續強化）。

## 8. 步驟 2：欄位配對

來源欄位 → JET 邏輯欄位的對應；自動建議（`mapping.autoSuggest`）與提交標準化（`mapping.commit.gl` / `mapping.commit.tb`）都在後端執行。GL 四種金額模式、TB 兩種變動模式（manifest Mapping 章節）。

草稿狀態提供兩種可切換的配對介面，由步驟頂部一個切換鈕選定：簡易清單與對照表格。預設是簡易清單；使用者選了哪一種，本次 app session 都記住（換專案保留，重開 app 回到預設簡易清單）。兩種介面是同一份配對草稿的不同視圖，共用同一份 `draft`，所以切換無損——在一種介面改到一半切到另一種，已選的對應都還在。

簡易清單：每個 JET 邏輯欄位一列，右側一個下拉去挑對應的來源欄。

對照表格：把配對動作做在資料本身上。畫面像一張試算表，每個來源欄是一直行，欄的標頭正上方放一個下拉，由它選定這欄對應到哪個 JET 邏輯欄位；標頭下方直接鋪該批次的前 10 列原貌，重用 `query.dataPreview`（GL 取 `glStaging`、TB 取 `tbStaging`），讓使用者過目幾筆值就能當場判斷這欄是什麼。這個模式採加寬版面，編輯區分兩欄：左邊是二維表，右邊是一條必填欄位鐵軌。鐵軌只列出目前模式的必填 JET 欄位，每個欄位未指派時顯示中性灰的「待指派」，指派後翻成淡綠的「✓」並標出對到的來源欄名；side/flag 模式的借方代碼字面值填了也算綠。全部必填變綠才啟用〔確認配對〕。

兩種介面共通的規則：下拉的可選欄位隨模式即時重算，而模式維持為表格上方的獨立選擇器，由它切模式。借方代碼是字面值不是來源欄，所以它在 side/flag 模式才以一個小輸入框出現。指派維持一對一：一個 JET 欄位至多對一個來源欄，一個來源欄至多對一個 JET 欄位。沒補齊必填就停用〔確認配對〕——這只是前端的引導，後端仍是權威。不論用哪一種介面，前端的正規狀態都是「欄位 → 來源欄」的 `draft` map，提交的 payload 與行為一字未變。

### 每個資料集的狀態模型

| 狀態 | 判定 | 呈現 |
|:---|:---|:---|
| 未匯入 | 無匯入批次 | 警示：請先完成「匯入資料」 |
| 草稿 | 尚未提交 | 頂部介面切換（簡易清單／對照表格）＋上方模式選擇器＋所選介面的編輯區（簡易清單＝每個 JET 欄位一列右側挑來源欄；對照表格＝來源欄為直行、每欄標頭一個下拉選定對應的 JET 欄位、標頭下方鋪該批次前 10 列原貌、右側必填鐵軌）＋（side/flag 模式才出現）借方代碼字面值輸入＋必填覆蓋檢核＋〔自動建議〕〔確認配對〕 |
| **已提交** | 已提交且草稿與快照一致 | **收合摘要卡**（淡綠）：「已提交，依此配對執行後續測試」＋模式、已標準化列數、提交時間＋**唯讀對照表**（來源欄為表頭、其下標對應的 JET 欄位、再附該批次前 10 列樣本；side/flag 模式另以小字補述借方代碼字面值）；動作只有〔重新配對〕〔預覽標準化資料〕——編輯表格與確認鈕**不渲染** |
| 草稿偏離 | 已提交但草稿/模式被改動（或按過〔重新配對〕） | 編輯表格＋黃色橫幅「下方修改尚未生效，目前仍以已提交版本執行」＋〔重新確認配對〕〔還原為已提交版本〕 |
| 來源變更失效 | 匯入動作清除了已提交配對 | 草稿畫面＋黃色橫幅「來源資料已變更，原配對已失效」 |

前端 committed 狀態保存完整快照 `{ projectedRowCount, committedUtc, mapping, mode }`（resume 自 `project.load.mapping`，標準化列數 resume 時未知則省略）。草稿態不再有獨立的「預覽來源資料」按鈕——來源原貌已內嵌在二維表的標頭下方。摘要卡的「預覽標準化資料」維持，換位到資料預覽的標準化後資料集。

## 9. 步驟 3：資料驗證與測試

兩張規則卡（資料驗證、風險預篩選），各有執行按鈕與上次執行時間；規則以「中文名＋一行說明＋狀態徽章」呈現，**不顯示代號**。狀態徽章：未執行＝灰、通過/未發現＝綠、命中（值得留意）＝黃、需要處理＝紅、彙總參考＝藍、無法執行＝虛線（tooltip 給 naReason）。

- 資料驗證（`validate.run`）：完整性測試、借貸不平測試、INF 抽樣測試、空值紀錄測試；頂部統計列（GL 筆數、傳票數、借貸淨額、查核期間）。空值紀錄在 UI 依異常類別拆成「核准日不在期間／空白摘要／空白傳票號碼·科目」三列呈現（「核准日不在期間」以核准日 `approval_date` 判定，2026-06-23 決策）（後端仍是單一 `null_records_test` 規則，前端按子計數與 `issues` 分列）。
- 風險預篩選（`prescreen.run`）：期末財報準備日後核准之分錄、分錄摘要出現特定描述、未預期出現之特定借貸組合（需科目配對）、分錄金額中有連續零的尾數（顯示本次門檻）、依分錄編製者彙總、較少使用之科目、週末過帳／核准、假日過帳／核准、摘要空白。
- counts 為摘要徽章，非 row list；明細分頁屬 `query.*Page` 里程碑。
- 結果自 `project.load.latestRuns` 原樣回放（schema v3 遷移後為 null，重按執行即恢復）。
- **「檢視」詳情展開**：有示警、命中或無法執行的項目，徽章旁會出現「檢視」切換；通過或未發現的項目不顯示切換。展開後在列下方顯示詳情，內容依項目而異：
  - 無法執行：顯示 `naReason` 說明。
  - 完整性、借貸不平、空值紀錄：顯示後端回傳的有界明細列（各 ≤50 筆）。
  - 預篩選命中：惰性呼叫 `filter.preview`（以該預篩選鍵組成單條件情境）取前 50 列，結果快取在前端、不重複請求。
  - 依編製者彙總、較少使用科目：直接顯示各自的彙總列表。

## 10. 步驟 4：進階條件篩選

Query Builder：條件群組（AND/OR 由左至右結合）組合篩選情境，前端只組裝 JSON、不評估規則。

條件型別依**審計意圖**分四組呈現（非資料格式；NN/g chunking）。「快速加入」每組一列（小標題＋該組條件鈕），型別下拉以同一套 optgroup 分塊。分組與標籤僅為前端呈現，型別的 wire 值（AST type）不變。

型別中文名一律取自 `jet-guide.md` §4 命名登錄表（如「自訂尾數位數」「科目配對分析」）；分組為前端呈現用，不改登錄名。

| 分組 | 條件型別 | 控制項 |
|:---|:---|:---|
| 風險預篩選訊號 | 預篩選 | row-tag 規則下拉（中文名；「未預期借貸組合」需科目配對、「非授權編製人員」需授權清單已匯入才出現） |
| 依欄位內容 | 文字條件 | 欄位＋關鍵字（逗號分隔）＋比對模式（包含/完全符合/排除） |
| 依欄位內容 | 金額區間 | 金額（絕對值）下限～上限（欄位標籤明示「不分借貸、以金額大小比較」） |
| 依欄位內容 | 日期區間 | 欄位（總帳日期/核准日）＋起迄 |
| 依欄位內容 | 自訂關鍵字 | 摘要含使用者關鍵字（逗號分隔） |
| 依分錄性質 | 期內/期外 | 總帳日期在查核期間內/外 |
| 依分錄性質 | 借貸限定 | 僅借方/僅貸方 |
| 依分錄性質 | 人工/自動 | 人工分錄/自動分錄 |
| 進階樣態分析 | 自訂尾數位數 | 尾數連續 0 位數（1–12） |
| 進階樣態分析 | 科目配對分析 | 三模式（精確配對/借方錨定/貸方錨定）＋借方/貸方標準化分類；需科目配對已匯入才提供 |
| 進階樣態分析 | 自訂編製人員張數 | 編製人員全期張數 ≤ N |
| 進階樣態分析 | 自訂科目張數 | 科目全期張數 ≤ N |

操作流程:先填情境名稱與篩選動機（兩者必填,會保留到工作底稿）;接著〔預覽這個情境〕,送 `filter.preview` 取得命中筆數、傳票數與前 50 列;最後〔保存為篩選情境〕,送 `filter.commit`（replace-all,至多 10 個）。

**必填驗證的呈現（全域樣式）。** 情境名稱、動機、或「至少一條件」未補齊時,前端在按下〔預覽〕/〔保存〕當下**就近**提示並不送出:未填欄位標紅框、欄位正下方一行紅字、按鈕上方一條彙總提示（`form-notice`）列出還缺什麼,並聚焦第一個有問題的欄位。提示一律落在主畫面,不依賴可收合的「狀態與訊息」欄。使用者補齊後即時放寬（不在打字途中責備）。這是前端引導,後端仍以 `invalid_scenario` 為權威。所用的 `JetUi.setFieldError / clearFieldError` 是全域 helper,其他步驟的必填表單可重用同一套呈現。已儲存的情境清單可展開詳情（條件 pill 摘要,並惰性以 `filter.preview` 載入該情境的命中預覽:命中筆數加前 10 筆 grid，結果快取不重抓）,也可移除。〔預覽測試母體〕換位到資料預覽的 GL 標準化後分錄。科目配對尚未匯入時,相關選項在前端鏡像隱藏;權威驗證仍在後端,以 `invalid_scenario` 把關。

## 11. 步驟 5：匯出底稿

佔位 stub（匯出里程碑）：屆時以保存的情境定義集合式重跑並以 OpenXML SAX 串流寫檔，分頁名用命名登錄表的中文規則名。

## 12. 資料契約備忘

- 所有後端呼叫一律經 `JetApi.*`；新增 action 順序：manifest → `SUPPORTED_ACTIONS` → handler → UI。
- Bridge payload/response 不搬運完整 GL/TB row 集合；預覽一律有界（≤50 列）。
- 固定 `data-bind` / `data-action` 識別字屬契約面，重命名前先確認 manifest 與測試。
- 規則 wire key（`completenessTest`、`postPeriodApproval`…）的前端鏡像集中在 `ui-core.js` 常數與 `validate-step.js` 的規則卡定義；改名一律 manifest 先行。
