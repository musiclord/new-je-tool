# JET 文件地圖與寫作規範

本檔說明這個 repository 的文件體系。它交代三件事：每份文件扮演什麼角色、什麼時候該更新它，以及寫文件時必須遵守哪些格式規範。任何 agent 或開發者在新增或修改文件之前，都先讀本檔。

## 文件體系

文件分三類。三類的角色不同，更新節奏也不同。

### 一、現況規格（living specification）

這類文件永遠反映 `main` 分支目前的實際行為。如果一次程式碼變更影響到這些文件所描述的內容，就必須在同一個 commit 裡一併更新它們。原因是：repository 是唯一的事實來源。任何寫在 repository 之外的決策（不論是在對話、聊天還是個人筆記裡）一律視為不存在。

| 文件 | 角色 |
|:---|:---|
| `README.md`(根目錄) | 專案識別、快速開始、目錄結構 |
| `AGENTS.md` | 跨 agent 邊界地圖:架構鐵律、契約先行流程、驗證指令 |
| `CLAUDE.md`、`.github/copilot-instructions.md` | 各 AI 工具的入口守則 |
| `docs/jet-guide.md` | 業務語意與架構的最深層權威:規則規格與命名登錄表(資料驗證、預篩選、進階篩選條件)、資料規模策略、資料庫 provider 策略、審計工作流程 |
| `docs/action-contract-manifest.md` | 前端 ↔ WebView2 bridge ↔ C# handler 之間所有 action 的 wire contract 唯一事實來源 |
| `docs/jet-frontend-description.md` | 前端結構、流程、狀態與資料契約的行為描述 |
| `docs/development-status.md` | 開發現況快照:目前可用的功能、進行中的工作、規劃中的方向、已擱置的決策 |

### 二、開發紀錄(development log)

`docs/development-log.md` 是依日期排列的開發紀錄，最新的放在最上面。每一輪開發寫成一個條目，記錄三件事：做了什麼、做了哪些關鍵決策、驗證結果如何。這份紀錄只增不改（append-only）。每個條目都寫成可以單獨讀懂的敘述，不假設讀者已經讀過其他條目。權威的細節仍然放在現況規格裡；這份紀錄只負責回溯「為什麼會變成現在這樣」。

### 三、設計提案與交接

| 文件 | 角色 |
|:---|:---|
| `docs/specs/YYYY-MM-DD-主題-design.md` | 設計提案。它是某一輪開發動工前定稿的設計快照。檔首標註目前「狀態」(設計中/實作中/已落地/已擱置)。實作落地之後,內容的權威性就轉移到現況規格,提案本身只剩歷史參考價值。等到 GUI 驗收通過、權威完全轉移到現況規格,依慣例就可以移除這份提案,歷史脈絡改由 `development-log.md` 承載。注意:這個目錄跟 `.claude/skills/` 下的 obra/superpowers skill 框架無關,它純粹是 JET 設計提案的存放處。 |
| `docs/windows-handoff.md` | 滾動式交接清單。內容是 macOS 開發端無法執行、必須在 Windows 環境才能完成的驗證任務。一張任務卡完成後,就移到「已完成」段。 |

## 寫作規範

以下規範整理自業界對 AI 協作專案文件的實務建議，重點有三項：現況規格要雙向回流、規格與變更要分離、repository 對 agent 要可讀。

1. **人類可讀的敘述優先。** 用完整句子說明，不要用只有作者自己看得懂的速記。一句話只講一個重點，不要把七八個子句和層層括號塞進同一句。技術名詞該保留就保留，但前後文要用自然的中文寫，不要生硬地中英夾雜。表格只放可以逐項列舉的事實，解釋一律寫在表格外的敘述裡。
2. **禁止流水代號。** 不要發明「M3」「W-2」「F1」「D4」這類流水編號去統稱里程碑、問題或決策。這種編號跨文件容易撞名，而且一旦離開原文就無法解讀。請改用描述性名稱，例如「多來源附加匯入」「CancellationToken 尚未串接」；需要排序時就用日期。規則名稱同樣不准用代號。早期的規則代號（V1–V4 代表驗證規則、R1–R8 代表預篩選規則、A2–A4 代表進階篩選條件）已於 2026-06-11 全面退役。退役原因是這些代號跨世代會產生歧義：legacy 時期的 V1 和後來重新編號的 V1 指的是不同的測試。現在規則一律用 `docs/jet-guide.md` §4 命名登錄表裡的具體名稱，例如完整性測試（`completeness_test`）、期末後核准（`post_period_approval`）等等。歷史代號只准出現在兩個地方：登錄表的「歷史代號」欄，以及對照 legacy 內容的敘述裡。
3. **領域名詞要精確。** 遇到模糊的概念，就用領域專有名詞把它講清楚（名詞清單見下方的速查表）。某個名詞第一次出現時，給出中文全名與英文原文，例如「總帳（General Ledger，GL）」。
4. **狀態要明確標示，而且不能誇大。** 用「規劃中／實作中／已落地／已擱置」這類明確說法，不要用「應該」「大概」這種含糊語氣去描述事實狀態。「已驗收」這個狀態只有在使用者親自做完 GUI 人工驗收之後才能標。如果程式已寫完、自動化測試全綠，但使用者還沒目視驗收，一律寫「已實作／已落地，待 GUI 驗收」，不准寫成「已驗收」。測試數字要據實回報實際的 passed、failed、skipped 筆數，不臆測，也不為了好看而美化。
5. **文件要跟程式碼放在同一個 commit 裡更新。** 做法是契約先行：要動 action、payload 或 workflow 行為之前，先改 manifest；改完落地之後，再回頭把受影響的現況規格一併更新。這樣雙向回流，可以避免規格與實作逐漸漂移。這一條約束的是變更的打包方式，也就是文件和程式碼不拆成兩個 commit；它並不是授權 agent 自行發動 commit。版控時機的規範見 `AGENTS.md` 的 Version Control 章節。
6. **交接任務要能自含。** `windows-handoff.md` 裡的每張任務卡，都必須讓另一端的 agent 不必讀對話紀錄就能直接執行。一張完整的任務卡要包含四項：指令、涵蓋內容、預期結果、失敗時的回報格式，缺一不可。

## 領域名詞速查

完整定義在 `docs/jet-guide.md`;這裡只列文件中高頻出現、容易被縮寫帶過的名詞:

| 名詞 | 意義 |
|:---|:---|
| 日記帳分錄測試(Journal Entry Testing,JET) | 本工具:依 ISA 240 / ISA 330 對管理階層凌駕控制風險做全母體分錄篩選 |
| 總帳(General Ledger,GL) | 受查者全年度的日記帳分錄明細,測試母體 |
| 試算表(Trial Balance,TB) | 科目層級的期初/期末餘額或變動,完整性測試的對照基準 |
| 傳票(voucher / journal entry document) | 分錄的憑證單位;一張傳票含多列分錄,借貸必須平衡(借貸不平測試的檢核對象) |
| 暫存表(staging table,`staging_*`) | 匯入檔案的原始列落地處,cell 一律存字串,不做業務解釋 |
| 正式表(target table,`target_*`) | 欄位配對提交後,由暫存列投影出的正規化資料;規則 SQL 只查這層 |
| 匯入批次(import batch) | 一次「資料集匯入」的單位:一個 GL 或 TB 資料集對應一個批次,可由多個來源檔/工作表組成 |
| 投影(projection) | 暫存列 → 正式列的轉換:金額以 `decimal` 解析後乘 `MoneyScale`(10000)轉 scaled BIGINT、日期正規化為 `"yyyy-MM-dd"` 字串。技術文件用「投影」;**UI 對使用者一律顯示「標準化」**(審計員語彙) |
| 欄位配對(field mapping) | 來源欄名 → 邏輯欄位(`docNum`、`postDate`…)的對應,提交時觸發投影 |
| 集合式 SQL(set-based SQL) | 規則計算的唯一允許形式:參數化 SQL 交給資料庫引擎,不得在 C# 對母體做 LINQ(guide §1.5.2) |
| 民國年(ROC calendar year) | 台灣常用紀年,西元 = 民國 + 1911;匯入日期正規化支援 `114/6/11` 與 `1140611` 形式(guide §3.1.3) |
| 受查者提供資料(Provided by Client,PBC) | 審計術語:受查者交付的原始檔案,格式不可控,是匯入強健化的對象 |
| 工作底稿(workpaper) | 審計留痕輸出:測試結果與篩選條件的匯出文件 |
| 資料庫提供者(database provider) | 同一套 Domain repository 介面下的可替換引擎:目前 SQLite,保留 SQL Server(雲端/大資料)與 DuckDB(本機分析)的更換縫(guide §13) |
