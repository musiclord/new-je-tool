# Windows 手動驗證清單

這是一份滾動式交接清單，只列「必須在實際 app（WinForms／WebView2）裡親手點過才能確認」的項目。自動化測試驅動不了 GUI，所以這些項目只能靠人工目視。自動化測試（`dotnet test`）由 agent 自跑，目前全綠。

新增任務卡時請寫成自含的一張卡，四個欄位缺一不可：啟動指令、涵蓋內容、預期結果、失敗回報格式。失敗回報要寫清楚三件事——在哪一步、做了什麼、看到什麼；若 F12 console 有錯，一併附上。共通的啟動指令是 `dotnet run --project src/JET/JET`（Debug 組建）；多數項目可先用「套用測試案件」按鈕備妥資料。某張卡驗收通過後，把它移到下方「已完成」段。

- **Step 4 進階篩選查詢建構器 UX 重整（2026-06-23 r3）—— 取代前一輪 KCT 面板的單條取代行為**

  - 涵蓋內容（Step 4「進階條件篩選」）：
    1. **無介面移位**：勾選/取消 KCT 卡、加入自訂條件、切換組合器時，「建立篩選情境」卡的欄位不應上下跳動；畫面上**不應**再有「建議名稱」藍框那一列。

    3. **自訂卡無 ×N**：自訂篩選條件卡只顯示「＋」與名稱，反覆點同一張不應出現 ×1/×2 之類計數變動；每點一下在最後一組新增一條可設定的條件。

  - 預期結果：如上。送後端的情境只含名稱/動機/條件群組（KCT 內部標記不外洩）。
  - 失敗回報格式：在哪一步、做了什麼、看到什麼（移位、兩態缺失、計數異常、組合器無效、報錯）；F12 console 有錯一併附上。

  - 5. **日期維度的非工作日週幾選擇器**：Step 1「日期維度」卡除了上傳假日與補班檔，還有一個一週七天（週一到週日）的勾選。使用者沒動過時，預設勾選週六與週日。改了勾選後即時保存，下次 `project.load` 回來時保留。這個設定會影響「週末過帳／核准」規則與週末篩選條件的判定。

- **WebView2 檔案對話框 reentrancy 崩潰修復（host 端）**
  - 啟動：`dotnet run --project src/JET/JET`（Debug）。
  - 背景：`host.selectFile`、`host.selectFiles`、`host.selectSavePath` 原本是在 WebView2 的 `WebMessageReceived` 事件處理常式裡**同步**開啟 `OpenFileDialog` 或 `SaveFileDialog`。這會觸發 WebView2 不支援的 reentrancy（事件處理常式還沒返回就重入），導致瀏覽器程序以 `0x80000003` 崩潰、整個 app 閃退，而且沒有任何 .NET 例外。修法是改走 `Form1.ShowDialogDeferredAsync`，用 `BeginInvoke` 把對話框延到事件處理常式返回之後再開。權威依據是 WebView2 的 Threading model › Reentrancy 文件，以及社群回報 WebView2Feedback #2946 與 #4648。
  - 涵蓋內容：對各個檔案對話框反覆操作，確認不再閃退。
    1. 「匯入資料」多來源精靈選**多個**檔案（`host.selectFiles`，.xlsx／.csv）。
    2. 行事曆假日／補班、科目配對、授權編製人員清單的**單檔**選擇（`host.selectFile`）。
    3. 「匯出底稿」的存檔對話框（`host.selectSavePath`）。
    4. 每個對話框各做幾次「開 → 選檔／取消」，包含快速連點，確認每次都正常返回、不凍結、不閃退。
  - 預期結果：對話框正常彈出與返回（選檔與取消都正確），選到的路徑或檔名照常進入後續流程。反覆操作不再閃退或凍結。F12 console 沒有 postMessage 例外。
  - 失敗回報格式：在哪個對話框、操作幾次後、看到什麼（凍結／閃退／錯誤）。若仍閃退，附上 `%LOCALAPPDATA%\JET\logs\` 最新一個 `.ndjson` 的末段，以及 Windows 事件檢視器裡 Application Error 的失敗模組與例外代碼。

- **D1 五處明細的「載入更多」（子專案 D1，全量明細分頁基礎設施；Task 6）**
  - 啟動：`dotnet run --project src/JET/JET`（Debug）。先在「資料驗證與測試」按「執行驗證」（或用「套用測試案件」），並在「進階條件篩選」保存至少一個會命中的情境。
  - 涵蓋內容：共五處，都要逐頁接續往下載。
    1. 「資料驗證與測試」→ 完整性測試〔檢視〕→ 表格下方「載入更多」逐頁接上科目差異列。
    2. 同一頁 → 借貸不平測試〔檢視〕→「載入更多」逐頁接上不平的傳票列。
    3. 同一頁 → INF 抽樣測試〔檢視〕→ 由空表開始，「載入更多」逐頁接上抽樣樣本的行層明細（借／貸、總帳日／核准日、建立／核准人員、摘要）。
    4. 同一頁 → 空值的三個子項（日期區間外／空白摘要／空白傳票號碼·科目）各自〔檢視〕→「載入更多」逐頁接上該類明細。其中「空白傳票號碼·科目」有兩顆鈕，傳票號與科目各自分頁。
    5. 「進階條件篩選」→ 已儲存情境〔詳情〕展開後的預覽（前 10 筆）下方「載入更多」→ 逐頁接上該情境命中的行層明細。
  - 預期結果：
    - **首次點擊就清掉上方的預覽列，改為一致排序的完整列表（這是關鍵驗收點）**。展開時上方顯示的是快速一瞥的預覽：完整性／借貸不平／空值是差額或筆數導向的 ≤50 筆樣本，篩選命中是前 10 筆。一旦點「載入更多」，這些預覽列應被清空，表格從第一頁重新接起（keyset 升冪，鍵為科目代碼／傳票號／分錄 id）。請確認接出來的列**不與原預覽列重複**，而且**整份是單一一致的升冪排序**，不是「預覽列加上另一種排序的接續列」混在一起。INF 抽樣本來就是空表展開，首次點擊就接第一頁，沒有預覽列要清。
    - 每點一次接一頁（≤200 列）到既有表尾；接到底時鈕自動消失；載入中時鈕顯示「載入中…」並 disabled，且不卡頓；大母體（例如約 140 萬列）逐頁載入要流暢，不應整頁水平爆版（寬表應在框內捲動）。
    - 「空白傳票號碼·科目」一格兩鈕的行為：先點任一鈕會清掉混合預覽並接上該類；再點另一鈕只接上另一類，**不應清掉前一鈕已載入的列**。兩類各自升冪、在同一表並陳、互不重複。
  - 失敗回報格式：在哪一處（上列 1–5）、做了什麼、看到什麼；F12 console 有錯一併附上。常見問題例如：接續列與預覽列重複、排序不一致、接續列不前進、接到底鈕沒消失、欄序錯位、postMessage 例外。

  - 啟動：`dotnet run --project src/JET/JET`（Debug）。先在「匯入資料」用「套用測試案件」備妥 GL/TB，並在「欄位配對」完成 GL/TB 配對。
  - 涵蓋內容：
